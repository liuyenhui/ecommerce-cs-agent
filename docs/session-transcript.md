# Session Transcript

本文件保存 2026-05-25 至 2026-05-26 关于电商客服 Agent 的启动讨论。内容为业务讨论整理版，不包含系统提示、工具输出和无关上下文。

## 用户原始方向

用户描述：

> 我们目前有客服系统，涉及多个电商，都是人工回复。可以 API 接入自动回复，但要智能。
> 每个平台的商品都有简介，但人工回复历史没有，现在起我们可以实现记录人工回复内容。
> 我们想搞一套平滑过渡的 Agent 回复客服问题，分 3 个阶段：
>
> 1. 先人工回复，但 Agent 给出回复内容候选。
> 2. 人工回复后，Agent 要学习人工回复的内容。
> 3. Agent 直接回复问题，遇到无法确定的问题再转人工。
>
> 最终效果：人工回复量越来越少，甚至不需要。

## 初始设计讨论结论

核心判断：

不要一开始追求 Agent 直接替代人工，而是建设一套可学习、可控、可回退的客服回复系统。目标不是一次到位替代人工，而是让人工回复逐渐变成训练数据和审核信号。

整体可以分 5 层：

1. 平台接入层：统一拼多多、淘宝、京东、抖音等客服消息格式。
2. 知识层：商品简介、商品规格、店铺规则、物流规则、售后政策、活动说明、人工回复沉淀。
3. Agent 决策层：判断意图、风险、知识命中、是否需要订单数据、置信度。
4. 回复生成层：生成候选回复、最终回复、补充问题或转人工原因。
5. 学习与评估层：记录 Agent 候选、人工最终回复、人工修改差异、用户后续反馈。

## 三阶段打法

### 阶段 1：人工主导，Agent 给候选回复

Agent 不直接发消息，只负责提高人工效率。

关键指标：

- 候选回复被采用率。
- 人工修改幅度。
- 平均回复时间。
- 转人工原因分布。

### 阶段 2：人工回复后，Agent 学习

这里的学习不建议一开始做模型微调。更稳的做法是先做知识沉淀：

- 人工回复归类。
- 去重。
- 抽象为问题类型、标准处理方式、可用话术、适用条件。
- 根据人工是否采用 Agent 候选来调整置信度。

### 阶段 3：Agent 自动回复，低置信转人工

适合自动回复：

- 商品参数。
- 发货时间。
- 物流状态。
- 常见售后。
- 活动规则。

不适合一开始自动回复：

- 投诉。
- 赔付。
- 退款争议。
- 平台处罚风险。
- 用户情绪强烈。
- 需要特殊承诺的问题。

## 重要数据资产

不要只存“问题和回答”，而要存：

- 买家问题。
- 商品信息。
- 订单状态。
- 平台。
- 店铺。
- Agent 候选。
- 人工最终回复。
- 人工编辑差异。
- 是否解决。
- 是否追问。
- 是否转售后、退款或投诉。

## 决策规则方向

推荐使用明确闸门，而不是让模型凭感觉决定：

- 高置信 + 低风险：自动回复。
- 中置信：给人工候选。
- 低置信或高风险：转人工。
- 需要订单数据但取不到：转人工或请求补充信息。
- 涉及承诺、赔付、投诉：默认人工确认。
