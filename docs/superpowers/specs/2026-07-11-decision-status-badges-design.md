# 决策状态 Badge 与节点导航优化设计

日期：2026-07-11
状态：用户已选择方案 A，待规格复核

## 背景

Customer Admin 的模拟咨询结果目前直接展示后端枚举值，例如 `candidate`、`low`。决策路径下方的节点选择器采用大块按钮排列，文字容易断行，视觉上像表格而不是流程导航。业务用户难以快速判断“AI 做了什么、风险如何、现在走到哪里”。

## 目标

1. 把消息卡片中的动作、风险和处理状态转换为中文业务标签。
2. 把决策路径下方的节点选择器改为紧凑、可换行的阶段 Badge。
3. 明确区分已完成、当前、等待、跳过和失败状态，不能只依赖颜色。
4. 保留节点点击、键盘焦点、节点详情联动和原始后端值，不修改 API 契约。

## 非目标

- 不修改 LangGraph 节点、边、运行状态或后端枚举。
- 不重做上方 X6 决策图。
- 不新增页面或改变 Customer Admin 与 System Admin 的站点边界。
- 不把完整技术状态重新暴露到聊天主界面。

## 方案

### 消息结果摘要

AI 回复正文下方使用最多三个独立 Badge：

- 动作：`candidate` → `建议回复`，`auto_reply` → `已自动回复`，`human_handoff` → `转人工处理`。
- 风险：`low` → `低风险`，`medium` → `中风险`，`high` → `高风险`。
- 状态：按现有决策状态映射为 `等待人工确认`、`等待补充资料`、`等待外部操作`、`处理完成` 或其他已有中文状态。

每个 Badge 通过 `title` 保留原始字段和值，方便技术排查。Badge 文本本身必须表达含义，颜色只作为辅助：动作使用蓝色、低风险/完成使用绿色、中风险/等待使用黄色、高风险/失败使用红色、未知或跳过使用灰色。

### 节点导航

节点选择器改为 `traceNodeBadges` 容器：

- 使用自然换行布局，不产生横向滚动。
- 每个节点为紧凑按钮，显示状态圆点、中文业务节点名和简短状态。
- 当前节点使用蓝色左侧强调线及“当前”文本；已完成节点显示“已完成”；等待、跳过、失败分别显示对应中文状态。
- 点击或键盘激活仍更新 `selectedNodeId`，并驱动原有节点详情区域。
- 在节点列表上方显示 `处理步骤` 和完成进度，例如 `7 / 8 已完成`。
- 原始节点 ID 和原始状态保留在 `title`，不进入主视觉文本。

### 响应式与可访问性

- 桌面端 Badge 高度保持紧凑，允许多行换行但不拆分单个 Badge 内的文字。
- 移动端节点 Badge 占用可用宽度，不隐藏节点，不依赖横向滚动。
- 所有可点击节点继续使用原生 `button`，具备可见焦点态和 `aria-pressed`。
- 状态必须同时有文字和颜色；对比度遵循现有 Customer Admin Carbon 风格 token。

## 数据与组件边界

- 在共享展示层集中维护动作、风险、决策状态和节点状态的本地化映射。
- Customer Admin 消息卡片复用共享的决策摘要展示函数，避免再次直接渲染后端枚举。
- `DecisionTraceReplay` 继续作为 Customer Admin 与 System Admin 共用组件；节点 Badge 改动同时覆盖两端回放，但不改变两端鉴权或数据来源。

## 测试

1. 展示单元测试：已知动作、风险、状态返回正确中文标签，未知值安全回退并保留原始值。
2. Trace 展示测试：节点状态本地化、完成进度和当前节点判定正确。
3. UI 回归守卫：聊天消息不再直接输出 `动作 candidate · 风险 low`；节点使用原生按钮、`aria-pressed` 和紧凑 Badge 类名。
4. 响应式守卫：移动端节点容器允许换行且不产生水平溢出。
5. 全量运行 TypeScript、Vitest、Admin 边界测试、UI 守卫、移动测试和 Customer/System 双构建。

## 验收标准

- 用户无需理解英文枚举即可判断回复方式、风险和当前处理状态。
- 决策路径底部不再出现大块、断行难看的表格式按钮。
- 当前节点、完成节点和异常节点仅凭文字也能区分。
- 点击任意节点仍能显示对应详情。
- Customer Admin 与 System Admin 的 API、会话和入口隔离保持不变。
